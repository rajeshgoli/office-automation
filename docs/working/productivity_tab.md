# Productivity Tab + History Enhancements

**Classification: Epic** — 3 sub-tickets, parallelizable after Part A lands.

## Context

This is an epic for adding a Productivity tab to the Android app, alongside two fixes to the existing History tab. It follows the same architecture established by the History tab (PR #21): backend endpoints in `src/orchestrator.py` with database queries in `src/database.py`, consumed by a Retrofit API service, repository, and ViewModel on the Android side, rendered with Compose Canvas charts.

The end goal is a **leverage ratio** — how much agent output (commits, lines, PRs) results from each unit of Rajesh's orchestration input (prompts, sessions, active time). No tool tracks this today. To get there, we need both sides of the equation: orchestrator input and agent output. This deliverable is the MVP — it captures the input side only. Phase 2 adds the output side and the ratio.

Rajesh doesn't write code. He orchestrates AI agents — Claude Code and OpenAI Codex — by sending them prompts. Every prompt is timestamped and logged in `~/.claude/history.jsonl` and `~/.codex/history.jsonl`. These files are the primary data source, not git stats. Git commits measure agent output; session history measures orchestrator input. The Productivity tab starts by showing orchestration patterns — when, where, how much — so that when Phase 2 lands the output metrics, the leverage view snaps together from data already flowing.

## Tenets

1. **Measure the orchestrator, not the agents.** The leverage ratio is the north star: `output / input`. This MVP captures the denominator — orchestration input. A day with 3 carefully-worded prompts that produced 200 commits has higher leverage than 50 prompts that went in circles. We start with volume and let Rajesh interpret; Phase 2 makes the ratio explicit.

2. **Reuse established patterns.** The History tab set the architecture: backend endpoint → Retrofit → Repository → ViewModel → Compose screen. The Productivity tab must follow the same patterns exactly. No new architectural concepts.

3. **Ship bundled.** The History tab has two known issues (missing temperature Y-axis labels, no door/window timeline). These ship in the same deliverable because they touch the same files and the door/window timeline contextualizes CO2 data that's already displayed.

## Scope

This deliverable has three parts:

**Part A — Data Pipeline (Mac Mini backend).** Sync session history files from the work Mac via rsync, parse them into an SQLite table, and serve them through new API endpoints. This runs entirely on the Mac Mini.

**Part B — Productivity Tab (Android).** A new third tab in the bottom navigation showing orchestration activity, project focus, and daily summary cards. Reuses the office sessions timeline from the History tab (moved, not duplicated).

**Part C — History Tab Fixes (Android + backend).** Temperature chart Y-axis labels and a door/window open-period timeline.

## Part A: Data Pipeline

### Source Files

Two tools produce session history:

**Claude Code** writes `~/.claude/history.jsonl` on the work Mac. Each line is a JSON object with exactly five fields:
```json
{"display":"the user prompt","pastedContents":{},"timestamp":1774676562599,"project":"/Users/rajesh/Desktop/automation/office-automate","sessionId":"b535d965-..."}
```
`timestamp` is epoch milliseconds. `project` is the absolute working directory path. `display` is the full text of what the user typed.

**Codex** writes `~/.codex/history.jsonl` on the work Mac. Each line:
```json
{"session_id":"019d30e7-...","ts":1774674087,"text":"ok PR the changes you made for crash fix"}
```
`ts` is epoch seconds (not milliseconds). `text` is the prompt. There is no project/cwd field — that lives in `~/.codex/state_5.sqlite`, in the `threads` table, keyed by `id` (which matches `session_id`). The `cwd` column holds the working directory.

### Filtering Rule

Messages generated by the session-manager system must be excluded. Human messages can also start with `[` — notably `[Pasted text #...]` (2,400+ occurrences in Claude history) and `[Image #...]` entries, which are real orchestration input. The filter must therefore match only known machine-generated prefixes, not a blanket `[`.

**Exclude a message if its text matches any of these prefixes.** This is the exhaustive list derived from the session-manager source code (`~/Desktop/automation/session-manager/src/`):

| Prefix | Source | Example |
|--------|--------|---------|
| `[Input from:` | `session_manager.py:2902` | `[Input from: spec-reviewer (d7436972) via sm send]` |
| `[sm job-watch]` | `message_queue.py:584+` | `[sm job-watch] Job completed: ...` |
| `[sm track]` | `message_queue.py:2371+` | `[sm track] Waiting on agent-1 ...` |
| `[sm remind]` | `message_queue.py:2404+` | `[sm remind]` |
| `[sm dispatch]` | `message_queue.py:3009+` | `[sm dispatch] Child agent update ...` |
| `[sm wait]` | `message_queue.py:3357+` | `[sm wait] Timeout reached ...` |
| `[sm context]` | `server.py:4699+` | `[sm context] Context usage at 50% ...` |
| `[sm info]` | `server.py:3572` | `[sm info] Uncommitted changes ...` |
| `[sm task-complete]` | `server.py:4268` | `[sm task-complete] Task finished` |
| `[sm] ` | `message_queue.py:1962+` | `[sm] Scheduled reminder: ...`, `[sm] scout-1535 completed` |

All session-manager injected messages start with either `[Input from:` or `[sm`. This yields a two-prefix check that covers every case:

```python
def is_machine_generated(text: str) -> bool:
    return text.startswith("[Input from:") or text.startswith("[sm")
```

This catches `[sm]`, `[sm `, `[sm remind]`, `[sm dispatch]`, etc. in a single prefix match.

**Include everything else**, including `[Pasted text #...]`, `[Image #...]`, and any other `[`-prefixed human input. Pasted content and images count as orchestration input — they represent the user composing a prompt with attachments.

If new session-manager message types are added, they must follow the `[sm` or `[Input from:` convention. The authoritative source is `~/Desktop/automation/session-manager/src/`.

### Sync Mechanism

A cron job on the Mac Mini rsyncs three files from the work Mac every 30 minutes:

```
rsync -az rajesh@<work-mac-ip>:~/.claude/history.jsonl  ~/office-automate/data/claude_history.jsonl
rsync -az rajesh@<work-mac-ip>:~/.codex/history.jsonl   ~/office-automate/data/codex_history.jsonl
rsync -az rajesh@<work-mac-ip>:~/.codex/state_5.sqlite  ~/office-automate/data/codex_state.sqlite
```

The work Mac's IP is statically reserved on the eero router (same pattern as the Mac Mini at `192.168.5.140`). SSH key auth is already configured between the machines (used by the occupancy detector deployment).

The `state_5.sqlite` file (~63 MB) is needed to resolve Codex session IDs to project paths. rsync's delta transfer keeps incremental syncs fast.

### Parser

A script `session_stats_parser.py` runs on the Mac Mini after each rsync. It reads both history files, filters out machine-generated messages by matching known sm prefixes, resolves Codex session IDs to project paths via the threads table, and upserts into the orchestrator's SQLite database.

**Parsing contract:**

Machine-generated message detection (shared by both tools):
```python
def is_machine_generated(text: str) -> bool:
    return text.startswith("[Input from:") or text.startswith("[sm")
```

For each record in `claude_history.jsonl`:
1. Parse JSON line. If the line is malformed (truncated by mid-sync rsync), skip it and log a warning.
2. Extract `display`, `timestamp`, `project`, `sessionId`.
3. If `is_machine_generated(display)`, skip.
4. Convert `timestamp` from epoch ms to local datetime: `datetime.fromtimestamp(timestamp / 1000)`.
5. Extract project name: `os.path.basename(project)` if project is non-empty, else `"unknown"`.
6. Emit row: `(local_datetime, "claude", project_name, session_id)`.

For each record in `codex_history.jsonl`:
1. Parse JSON line. If malformed, skip and log a warning.
2. Extract `text`, `ts`, `session_id`.
3. If `is_machine_generated(text)`, skip.
4. Convert `ts` from epoch seconds to local datetime: `datetime.fromtimestamp(ts)`.
5. Look up `session_id` in `codex_state.sqlite` → `threads.cwd`. Extract basename. If not found, use `"unknown"`.
6. Emit row: `(local_datetime, "codex", project_name, session_id)`.

**Idempotency:** The parser must track the last-processed line count for each file (stored in a small metadata table or file). On each run, it reads only new lines appended since the last run. Both history.jsonl files are append-only — lines are never modified or deleted.

### Database Schema

Add to `src/database.py`, following the existing pattern of `CREATE TABLE IF NOT EXISTS` in `_init_schema()` (line 48):

```sql
CREATE TABLE IF NOT EXISTS orchestration_activity (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp DATETIME NOT NULL,
    tool TEXT NOT NULL CHECK(tool IN ('claude', 'codex')),
    project TEXT NOT NULL DEFAULT 'unknown',
    session_id TEXT NOT NULL
);
CREATE INDEX IF NOT EXISTS idx_orch_timestamp ON orchestration_activity(timestamp);
CREATE INDEX IF NOT EXISTS idx_orch_date ON orchestration_activity(date(timestamp));
```

All timestamps are local time (PST), consistent with every other table in this database (see CLAUDE.md: "All timestamps are stored in local time (PST), not UTC").

### API Endpoints

Add to `src/orchestrator.py`, registered alongside the existing `/history/*` routes at lines 1991-1994.

**`GET /history/orchestration?days=7`**

Returns per-day orchestration data for the timeline and summary cards.

Response:
```json
{
  "ok": true,
  "days": [
    {
      "date": "2026-03-27",
      "messages": 42,
      "sessions": 5,
      "first_prompt": "09:15",
      "last_prompt": "18:30",
      "by_tool": {"claude": 35, "codex": 7},
      "timestamps": [
        {"time": "09:15", "tool": "claude"},
        {"time": "09:18", "tool": "claude"},
        {"time": "10:02", "tool": "codex"}
      ]
    }
  ]
}
```

The `timestamps` array contains every human prompt's HH:MM and tool, used to render the orchestration activity timeline. The rest are aggregates for the summary cards.

**`GET /history/project-focus?days=7`**

Returns per-day per-project message counts for the stacked bar chart.

Response:
```json
{
  "ok": true,
  "days": [
    {
      "date": "2026-03-27",
      "total": 42,
      "projects": [
        {"name": "fractal-market-simulator", "messages": 25},
        {"name": "office-automate", "messages": 12},
        {"name": "taskbar", "messages": 5}
      ]
    }
  ]
}
```

Projects are sorted by message count descending within each day.

**`GET /history/openings?days=7`** (for Part C — door/window timeline)

Queries `device_events` for door and window open/close pairs. An "opening" is defined as an `open` event followed by the next `closed` event for the same `device_type`. Three edge cases:

1. **Unclosed opening:** If the most recent event is `open` with no subsequent `closed`, the opening extends to the current time (today) or end of day (past days).
2. **Cross-boundary opening:** If a door/window was opened before the queried date range began and closed during it, the query must seed initial state by reading the latest event before the cutoff. This is the same pattern used by `get_daily_stats()` in `src/database.py` for occupancy/HVAC/ERV state seeding. Without this, a door opened at 23:00 and closed at 09:00 the next day would render nothing on the next day's timeline.
3. **Midnight-spanning opening:** An opening that starts on day N and closes on day N+1 must appear in both days — extending to 23:59 on day N and starting at 00:00 on day N+1.

Response:
```json
{
  "ok": true,
  "days": [
    {
      "date": "2026-03-27",
      "door": [
        {"open": "09:12", "close": "09:14"},
        {"open": "12:30", "close": "12:35"}
      ],
      "window": [
        {"open": "14:00", "close": "16:30"}
      ]
    }
  ]
}
```

## Part C: History Tab Fixes

### Fix: Temperature Chart Y-axis Labels

The `TemperatureChart` composable (`HistoryScreen.kt` lines 665-724) draws a line chart with no Y-axis labels or grid lines. The `CandlestickChart` composable (lines 442-539) has both. Port the Y-axis pattern from `CandlestickChart` to `TemperatureChart`:

1. Add `leftPadding = 40.dp` to reserve space for labels (matches `CandlestickChart` line 443).
2. Calculate grid step based on temperature range. Temperature ranges are small (typically 60-80 F), so use steps of 5 F.
3. Draw horizontal grid lines with the same `Border` color and 0.3f alpha used in `CandlestickChart`.
4. Draw labels using `textMeasurer.measure()` with the same `FontFamily.Monospace, 8sp` font used in `CandlestickChart`. Format as integer F values (e.g., "65", "70", "75").

The min/max band and orange line rendering remain unchanged — they just shift right by `leftPadding`.

## Out of Scope

- **Agent leverage metrics (Phase 2).** No git stats collection, no GitHub API integration, no leverage ratio calculation. This deliverable only tracks orchestration input.
- **Claude session-meta parsing.** The `usage-data/session-meta/*.json` files contain rich per-session analytics (tokens, tool counts, lines changed). Valuable for Phase 2 leverage, but not needed for Phase 1.
- **Real-time updates.** The data pipeline runs every 30 minutes via cron. No WebSocket push for new prompts. The Android app fetches on tab open, same as the History tab.
- **Author attribution.** No attempt to distinguish Rajesh's commits from agent commits. Deferred to Phase 2.
- **Codex rollout transcript parsing.** The full conversation logs in `~/.codex/sessions/` are not parsed. Only the top-level `history.jsonl` is used.
